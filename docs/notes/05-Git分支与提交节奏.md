# Git 分支与提交节奏

## 一、这次看到的现象

做完 `app.js` 事件总线和 `bubble.js` 气泡拆分后，代码已经完成了一个清楚的小目标。

这时遇到的问题是：

```text
什么时候应该提交？
什么时候应该开新分支？
分支名应该怎么取？
```

这些问题不是语法问题，而是项目协作和版本管理问题。

## 二、什么时候适合提交

提交可以理解成一次清楚的存档。

适合提交的情况：

```text
完成了一个小功能
完成了一次小重构
修复了一个明确 bug
新增或更新了学习笔记
已经做过基本验证
可以用一句话说明这次改动目的
```

不太适合提交的情况：

```text
代码还跑不起来
文件只改到一半
自己还说不清这次改动目的
一次改动混入了多个不相关目标
```

一个好提交应该像这样：

```text
引入事件总线并拆分气泡模块
```

它有明确主题，也方便以后回看历史。

## 三、什么时候适合开新分支

分支可以理解成一条独立的实验线。

适合开新分支的情况：

```text
准备做一个新功能
准备做一次可能影响较大的重构
想保留主分支稳定
这项工作可能需要多次提交
后续可能要发 Pull Request
```

例如：

```text
设置面板
CC 告警
宠物状态机
托盘菜单
```

这些都适合单独开分支。

## 四、分支名怎么取

分支名应该短、清楚、说明主题。

例如：

```text
event-bus
settings-panel
cc-alerts
pet-state-machine
tray-menu
```

这次把 `develop` 改名为 `event-bus`，原因是当前这条分支的实际主题已经变成：

```text
引入事件总线，并把部分交互迁移到事件驱动。
```

`event-bus` 比 `develop` 更能说明这条分支在做什么。

## 五、提交前检查清单

提交前可以快速问自己：

```text
1. 这次改动能不能用一句话说明？
2. 有没有运行必要的语法检查？
3. 应用有没有做过基本启动验证？
4. 有没有混入不相关文件？
5. 重要知识点有没有写进 docs/notes？
```

如果大部分答案都是“是”，就可以提交。

## 六、本项目推荐节奏

这个项目是学习型项目，推荐节奏是：

```text
一个学习主题或功能 -> 一个分支
一个清楚的小目标 -> 一个提交
重要知识点 -> 一篇学习笔记
```

例如这次：

```text
分支：event-bus
目标：引入事件总线并拆分气泡模块
提交：♻️ 引入事件总线并拆分气泡模块
笔记：04-暴露与事件总线.md
```

## 七、这次学到的通用知识点

```text
现象：完成一个小重构后，不确定是否应该提交。
原因：提交不是按代码行数判断，而是按“是否完成了一个可解释的小目标”判断。
知识点：Git 提交是项目历史里的清楚存档；分支是围绕一个主题展开的独立工作线。
```

## 八、新项目的 main 应该先放什么

`main` 表示当前已经确认、可以交给别人使用或继续开发的稳定基线。稳定不等于功能很多，而是已有内容处于自洽状态。

空仓库建立后，应先在 `main` 放入最小项目基线：

```text
README.md：项目目标、技术栈、安装和启动方法
.gitignore：node_modules、data、日志、密钥和临时文件
package.json：依赖和最小启动命令
最小代码骨架：如果技术栈已经确定，至少能够正常启动
AGENTS.md 或 CONTRIBUTING.md：协作、测试和远程操作规则
docs/specs：已经确认的产品或架构方向
LICENSE：公开项目需要时添加
```

如果项目仍处于纯设计阶段，`main` 可以只有说明、忽略规则和设计文档，但 README 应明确“当前尚未提供可运行程序”。一旦已经加入代码，`main` 应尽量保持 `npm install` 和 `npm start` 能正常工作。

初始化阶段不应提交：

```text
node_modules
运行时快照和日志
API Key、Token 或其他凭据
用户级 Claude Code 配置
只适用于本机的绝对路径
做到一半且无法运行的复杂功能
```

个人新仓库第一次建立基线时，可以直接提交到 `main`。当 `main` 已经成为共享稳定版本后，产品功能和 Bug 修复原则上都通过分支和 Pull Request 进入。

## 九、从 Issue 开始一个功能分支

准备处理新目标前，先查看 GitHub Issue 的现象、复现步骤、预期行为和验收标准。一个分支最好对应一个 Issue，或者一个可以独立验收的小目标。

开始前先同步稳定基线：

```bash
git switch main
git pull --ff-only
git switch -c cc-alerts
```

好的分支主题可以用一句话说明：

```text
基于上下文使用率显示阈值告警
让透明窗口的空白区域支持点击穿透
支持配置并保存监控项目路径
```

下面这种目标太宽泛：

```text
继续开发桌宠
完善所有功能
修复一些问题
```

一个功能分支可以包含多个相关提交，但不应混入可以独立验收的其他功能。例如 `cc-alerts` 可以同时包含状态判断、测试和学习笔记，但不应顺便加入多显示器拖动或设置面板。

## 十、什么时候推送和创建 Draft PR

本地完成第一个有意义的提交后，就可以推送分支。推送不表示功能已经完成，它还承担远程备份、协作查看和触发 CI 的作用。

```bash
git push -u origin cc-alerts
```

分支已经有明确目标并推送后，应尽早创建 Draft PR，而不是等所有工作结束才创建。

Draft PR 表达：

```text
方向已经明确，可以查看当前进度，但现在还不能合并。
```

PR 描述可以使用清单持续记录进度：

```markdown
## 目标

Closes #7

## 当前进度

- [x] 定义 normal、warning 和 danger
- [ ] 防止同一区间重复提醒
- [ ] 补充多会话测试
- [ ] 完成 Electron UI 验收
```

尽早创建 Draft PR 可以提前建立 Issue、代码和验收标准之间的关联，也能更早发现分支范围是否正在变得过大。

## 十一、Draft 什么时候转为 Ready

Ready for review 表示实现者认为功能已经完成，可以正式审阅并准备合并。

转为 Ready 前应确认：

```text
Issue 验收标准已经逐项完成
自动化测试和语法检查通过
必要的 UI 手动测试已经完成
临时调试代码和测试数据已经清理
PR 描述与实际实现一致
没有明知未完成的核心功能
工作区干净且最新提交已经推送
分支与 main 没有冲突
```

Draft 阶段允许存在尚未完成的清单；Ready 阶段仍可能在审阅中发现问题，但不应把明显未完成的主体功能交给审阅者收尾。

## 十二、什么时候计划合并

计划合并应发生在 PR 转为 Ready 之后。合并前需要同时满足三类条件。

技术条件：

```text
测试通过
没有合并冲突
没有已知阻塞 Bug
代码结构符合项目约定
没有敏感数据或无关文件
```

产品条件：

```text
Issue 验收标准完成
用户可见行为符合预期
必要的人工测试已经完成
文档与代码保持一致
```

协作条件：

```text
PR 已经 Ready
未解决的审阅意见已经处理
合并目标和影响范围明确
已经取得需要的用户确认
```

推荐顺序：

```text
功能实现完成
    -> 本地回归通过
    -> Draft 转 Ready
    -> 正式审阅
    -> 处理审阅意见
    -> 再次验证
    -> 合并
```

## 十三、选择哪种合并方式

### Squash and merge

把一个 PR 压成 `main` 上的一个提交。适合提交较碎、希望主分支历史简洁的小功能分支。

未来一个 Issue 一个短分支时，本项目可以优先使用 Squash merge。

### Merge commit

保留分支中的所有提交，并增加一个合并提交。适合分支内多个提交都具有独立学习或设计意义的阶段性版本。

当前 `event-bus` 包含 MVP、事件总线、statusLine、告警和学习文档，如果需要保留完整学习历史，可以使用 Merge commit。

### Rebase and merge

把分支提交依次放到 `main` 顶部，不产生合并提交。历史线性，但会改变提交在历史中的位置。当前学习型项目不必优先使用。

## 十四、合并后为什么可以删除功能分支

分支不是一份独立代码副本，而是指向某个提交的名字。

合并前：

```text
main
  \
   cc-alerts -> A -> B -> C
```

合并后，`main` 已经包含这些改动：

```text
main -> A -> B -> C -> 合并结果
cc-alerts -> C
```

此时删除 `cc-alerts` 只是删除指向提交 C 的分支名称，不会删除已经进入 `main` 的代码和历史。

正确的清理顺序是：

```text
PR 合并
    -> 确认 main 包含改动
    -> 确认关联 Issue 已关闭
    -> 同步本地 main
    -> 删除远程功能分支
    -> 删除本地功能分支
```

同步并安全删除本地分支：

```bash
git switch main
git pull --ff-only
git branch -d cc-alerts
```

删除远程分支：

```bash
git push origin --delete cc-alerts
```

`git branch -d` 会在分支尚未合并时拒绝删除。不要随意使用强制删除 `git branch -D`，除非已经明确确认分支中的独有提交不再需要。

删除远程分支属于 GitHub 远程写操作，当前项目要求执行前取得用户确认。

## 十五、没有合并但决定放弃分支

如果方案被其他实现取代，推荐：

1. 在 PR 说明放弃原因或替代 Issue；
2. 关闭 PR，但不合并；
3. 确认分支中的独有提交不再需要；
4. 必要时把有价值的提交迁移到新分支；
5. 删除远程和本地旧分支。

如果还不确定是否需要旧提交，可以暂时保留分支，或者创建明确的备份标签，不应为了整理列表而强制删除尚未确认的工作。

## 十六、本项目推荐的完整工作流

```text
GitHub Issue 定义问题和验收标准
    -> 从最新 main 创建短功能分支
    -> 完成一个可解释的小步骤
    -> 本地验证并提交
    -> 确认后推送
    -> 尽早创建 Draft PR
    -> 继续小步开发、测试和更新文档
    -> 完成 Issue 验收标准
    -> 自动化测试和必要的 UI 验收
    -> 清理测试数据
    -> Draft 转 Ready
    -> 正式审阅并处理意见
    -> 再次验证
    -> 确认后合并到 main
    -> Issue 自动关闭
    -> 同步 main
    -> 确认后删除远程功能分支
    -> 删除本地功能分支
    -> 从新的 main 创建下一条分支
```

小型个人项目长期保留一个 `main` 已经足够，不需要因为习惯而额外维护 `develop`、`staging` 等长期分支。

## 十七、当前 event-bus 分支的判断

当前 `event-bus` 已经不只是事件总线功能，而是包含：

```text
桌宠 MVP
事件总线和 UI 模块拆分
Claude Code statusLine 数据桥接
监控恢复和安全降级
上下文阈值告警
设计文档和学习笔记
```

它属于阶段性基线 PR，而不是理想的小功能 PR。当前不需要重写历史或强行拆成多个旧 PR，更合理的做法是：

1. 把 PR #10 定位为项目第一个可运行阶段版本；
2. 对整个分支进行基线审阅；
3. 转为 Ready 后正式审阅；
4. 合并到 `main`，让 `main` 成为新的稳定基线；
5. 确认 Issue #7 自动关闭；
6. 删除 `event-bus`；
7. 后续每个 Issue 从最新 `main` 创建独立短分支；
8. 尽早创建 Draft PR，避免再次积累成超大分支。

## 十八、五条最简判断规则

```text
1. 空仓库先在 main 建立最小、自洽的项目基线。
2. main 成为共享稳定版本后，不直接开发产品功能。
3. 每个独立目标从最新 main 创建短分支。
4. 分支有明确方向后尽早创建 Draft PR。
5. 验收完成后转 Ready，审阅通过再合并；确认 main 正常后删除功能分支。
```

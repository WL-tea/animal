# Git 分支与提交节奏

## 一、这次看到的现象

做完 `app.js` 事件总线和 `bubble.js` 气泡拆分后，代码已经完成了一个清楚的小目标。

这时遇到的问题是：

```text
什么时候应该提交？
什么时候应该开新分支？
分支名应该怎么取？
```

这些问题不是语法问题，而是项目协作和版本管理问题。

## 二、什么时候适合提交

提交可以理解成一次清楚的存档。

适合提交的情况：

```text
完成了一个小功能
完成了一次小重构
修复了一个明确 bug
新增或更新了学习笔记
已经做过基本验证
可以用一句话说明这次改动目的
```

不太适合提交的情况：

```text
代码还跑不起来
文件只改到一半
自己还说不清这次改动目的
一次改动混入了多个不相关目标
```

一个好提交应该像这样：

```text
引入事件总线并拆分气泡模块
```

它有明确主题，也方便以后回看历史。

## 三、什么时候适合开新分支

分支可以理解成一条独立的实验线。

适合开新分支的情况：

```text
准备做一个新功能
准备做一次可能影响较大的重构
想保留主分支稳定
这项工作可能需要多次提交
后续可能要发 Pull Request
```

例如：

```text
设置面板
CC 告警
宠物状态机
托盘菜单
```

这些都适合单独开分支。

## 四、分支名怎么取

分支名应该短、清楚、说明主题。

例如：

```text
event-bus
settings-panel
cc-alerts
pet-state-machine
tray-menu
```

这次把 `develop` 改名为 `event-bus`，原因是当前这条分支的实际主题已经变成：

```text
引入事件总线，并把部分交互迁移到事件驱动。
```

`event-bus` 比 `develop` 更能说明这条分支在做什么。

## 五、提交前检查清单

提交前可以快速问自己：

```text
1. 这次改动能不能用一句话说明？
2. 有没有运行必要的语法检查？
3. 应用有没有做过基本启动验证？
4. 有没有混入不相关文件？
5. 重要知识点有没有写进 docs/notes？
```

如果大部分答案都是“是”，就可以提交。

## 六、本项目推荐节奏

这个项目是学习型项目，推荐节奏是：

```text
一个学习主题或功能 -> 一个分支
一个清楚的小目标 -> 一个提交
重要知识点 -> 一篇学习笔记
```

例如这次：

```text
分支：event-bus
目标：引入事件总线并拆分气泡模块
提交：♻️ 引入事件总线并拆分气泡模块
笔记：04-暴露与事件总线.md
```

## 七、这次学到的通用知识点

```text
现象：完成一个小重构后，不确定是否应该提交。
原因：提交不是按代码行数判断，而是按“是否完成了一个可解释的小目标”判断。
知识点：Git 提交是项目历史里的清楚存档；分支是围绕一个主题展开的独立工作线。
```

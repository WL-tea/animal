# CC 数据入口与单向数据流

## 一、这次看到的现象

详情面板需要显示 Claude Code 数据，所以最早的写法是让 `detail.js` 直接监听：

```text
window.ccAPI.onUpdate(...)
window.ccAPI.refresh()
```

这个写法能跑，但它让 `detail.js` 同时承担了两个职责：

```text
接收外部 CC 数据
渲染详情面板 UI
```

后面如果气泡、宠物状态、设置面板也要使用同一份 CC 数据，它们可能也会各自去监听 `window.ccAPI`。这样数据入口会散落在多个 UI 模块里。

## 二、真正原因是什么

`window.ccAPI` 是 Electron preload 暴露给页面的外部能力。它更像“数据管道入口”，不属于某一个具体 UI 面板。

如果让某个 UI 文件直接接管这个入口，短期简单，长期会出现两个问题：

```text
数据来源不集中：不知道最新 CC 数据到底由谁保存
模块职责混乱：UI 模块既要管数据接入，又要管 DOM 渲染
```

所以这次把入口移动到 `app.js`：

```text
cc-monitor -> preload -> app.js -> detail.js
```

`app.js` 负责接收和保存最新数据，然后发出 `cc:update` 事件。`detail.js` 只监听这个事件，并把数据渲染到详情面板。

## 三、学到的通用知识点

单向数据流的核心不是“代码绕远路”，而是让每个模块只关心自己的职责：

```text
preload.js：安全暴露主进程能力
app.js：保存页面级状态，转发公共事件
detail.js：监听数据变化，渲染详情面板
```

这带来的好处是，后续新增气泡提醒或宠物状态时，不需要再去改 `cc-monitor` 或 `preload` 的数据入口，只需要监听同一个事件：

```js
window.petApp.on("cc:update", (data) => {
    // 根据最新 CC 数据做自己的事
});
```

这就是事件总线和单向数据流配合使用的意义：数据从一个入口进入页面，再由页面主控分发给各个 UI 模块。

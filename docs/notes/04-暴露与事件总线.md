# 暴露与事件总线

## 一、这次看到的现象

项目里后续会继续增加气泡、详情面板、设置面板、宠物状态机和 CC 告警。

如果每个模块直接调用其他模块里的函数，短期能跑，但功能变多后会出现一个问题：

```text
一个模块需要知道太多别的模块细节。
```

例如：

```text
pet.js 双击宠物 -> 直接调用 detail.js 里的 showDetail()
detail.js 收到 CC 数据 -> 以后可能还要通知 bubble.js 和 pet.js
settings.js 修改阈值 -> 以后可能还要影响 detail.js、bubble.js、pet.js
```

这样会让模块关系越来越乱。

## 二、什么是暴露

“暴露”就是：

```text
把原本只在一个地方能访问的东西，放到另一个地方也能访问的位置。
```

普通浏览器页面里，常见的公共位置是 `window`。

```js
const petApp = {};

window.petApp = petApp;
```

这句的意思是：

```text
把 petApp 暴露到 window.petApp 上。
```

这样其他 JS 文件就可以访问：

```js
window.petApp
```

项目里还有一个 Electron 的例子：

```js
contextBridge.exposeInMainWorld("ccAPI", {
    refresh: () => ipcRenderer.invoke("cc-refresh"),
});
```

它的意思是：

```text
把安全筛选过的 ccAPI 暴露给页面使用。
```

区别是：

```text
window.petApp：页面内部 JS 模块之间共享。
window.ccAPI：Electron preload 把主进程能力安全地交给页面。
```

暴露能力要克制。暴露越多，其他代码能碰到的东西越多，后续越难维护。

## 三、什么是事件总线

事件总线可以理解成一个“中转站”。

模块不直接互相调用，而是：

```text
一个模块发出事件。
另一个模块监听事件。
```

例如：

```js
window.petApp.emit("detail:open");
```

意思是：

```text
有人请求打开详情面板。
```

另一个模块可以监听：

```js
window.petApp.on("detail:open", () => {
    showDetail();
});
```

这样 `pet.js` 不需要知道 `detail.js` 里有没有 `showDetail()`，它只需要发出一个事件。

## 四、为什么本项目适合引入 app.js

这个桌宠项目后面会有多个模块：

```text
pet.js：宠物移动、拖拽、状态
bubble.js：气泡显示
detail.js：详情面板
settings.js：设置面板
cc-monitor.js：读取 Claude Code 数据
```

这些模块会互相影响，但不应该互相直接调用。

目标数据流是：

```text
cc-monitor -> preload -> app.js -> 各 UI 模块
```

`app.js` 的角色是：

```text
接收外部数据
保存页面级状态
转发事件
```

它不应该负责：

```text
不画宠物
不拼详情 HTML
不写气泡样式
不处理具体动画
```

如果 `app.js` 开始塞进太多具体 UI 代码，它也会变乱。

## 五、什么时候不需要事件总线

事件总线不是所有项目都必须用。

可以这样判断：

```text
一对一、很简单：直接调用。
一对多、跨模块：考虑事件。
功能还很少：先直接写，跑通后再整理。
```

例如一个按钮只负责打开一个面板，直接写也很好：

```js
button.addEventListener("click", () => {
    panel.hidden = false;
});
```

但如果一个数据更新后会影响详情、气泡、宠物状态和设置面板，就适合交给 `app.js` 统一转发。

## 六、这次改动学到的通用知识点

```text
现象：模块之间直接调用很容易写出第一版。
原因：直接调用最简单，但模块会越来越互相依赖。
知识点：事件总线可以减少模块之间的直接依赖，让模块只关心自己监听的事件。
```

# 多窗口职责与独立置顶

## 1. 为什么详情面板需要独立窗口

原来的桌宠、气泡和详情面板都位于同一个 `BrowserWindow`。CSS 可以改变同一窗口内部元素的布局，却不能让其中一部分永久置顶、另一部分按普通窗口层级工作。

`alwaysOnTop`、焦点、系统拖动和多显示器坐标都属于操作系统窗口能力，它们的控制边界是 `BrowserWindow`。因此本次把界面拆成：

- 宠物窗口：透明、无边框、固定大小，可以由用户决定是否始终置顶；
- 详情窗口：不透明、无边框、可调整大小，打开时到前方，之后遵守普通窗口层级；
- 设置面板：仍是详情窗口内部的渲染层面板，不额外创建第三个系统窗口。

## 2. 一个监控器如何服务两个窗口

文件监听器应该由主进程拥有，不能因为打开一个详情窗口就再创建一套监听器。否则会出现重复读取、重复事件、重复告警和难以清理的资源。

当前结构是：

```text
一个 CCMonitor
    -> 主进程广播目标
        -> 宠物窗口 webContents（告警气泡）
        -> 详情窗口 webContents（详情渲染）
```

广播前必须检查窗口和 `webContents` 是否已经销毁。详情窗口关闭后从广播目标中移除，但监控器继续为宠物窗口工作；只有宠物窗口关闭或应用退出时才停止监听。

## 3. 跨窗口操作为什么需要 IPC

渲染页面不能直接获得 `BrowserWindow`，否则会破坏 `contextIsolation` 和最小权限原则。宠物双击和详情关闭分别通过固定方法请求主进程：

```text
pet.js -> window.windowAPI.openDetail()
detail.js -> window.windowAPI.closeDetail()
```

`preload.js` 只暴露这两个明确动作，页面不能传入任意 IPC channel，也不能直接访问 Electron 或 Node.js。

## 4. 无边框窗口的拖动区域

去掉系统边框后，窗口不再自带标题栏，需要在 CSS 中声明拖动区域：

```css
.titlebar {
    app-region: drag;
    -webkit-app-region: drag;
    user-select: none;
}

.titlebar button {
    app-region: no-drag;
    -webkit-app-region: no-drag;
}
```

拖动区域会把鼠标操作交给系统窗口移动，因此按钮必须标记为 `no-drag`，否则关闭按钮会无法点击。当前同时保留标准写法和带前缀写法，以兼容项目使用的 Electron 版本。

显示器断开或排列改变时，主进程会检查详情窗口与所有显示器工作区的重叠面积。如果没有保留足够的可抓取区域，就把详情窗口恢复到主显示器中央。

## 5. 设置版本迁移与合并保存

设置结构从版本 1 升级为版本 2：

```json
{
  "version": 2,
  "projects": [],
  "petAlwaysOnTop": true
}
```

旧配置没有 `petAlwaysOnTop` 时自动使用 `true`，因此升级不会改变原有行为。写入项目列表时必须先读取现有设置再合并 `projects`，否则只保存项目字段会把置顶选项重置为默认值。

设置变化的数据流是：

```text
settings.js
    -> preload.js 固定 API
    -> ipcMain handler
    -> settings-store.js 原子保存
    -> petWindow.setAlwaysOnTop(...)
```

这样既保证修改立即生效，也保证应用重启后能够恢复。

## 6. 生命周期检查清单

- 重复双击宠物只复用并唤回现有详情窗口；
- 关闭详情窗口不会停止监控或关闭宠物；
- 关闭宠物窗口时同时关闭详情，并只停止一次监控器；
- 新详情窗口加载后主动请求刷新，避免错过创建之前的数据；
- 所有 IPC 都保持固定方法与固定数据类型；
- 项目增删不会覆盖其他设置字段。

# 多窗口职责与独立置顶

## 1. 为什么详情面板需要独立窗口

原来的桌宠、气泡和详情面板都位于同一个 `BrowserWindow`。CSS 可以改变同一窗口内部元素的布局，却不能让其中一部分永久置顶、另一部分按普通窗口层级工作。

`alwaysOnTop`、焦点、系统拖动和多显示器坐标都属于操作系统窗口能力，它们的控制边界是 `BrowserWindow`。因此本次把界面拆成：

- 宠物窗口：透明、无边框、固定大小，可以由用户决定是否始终置顶；
- 详情窗口：不透明、无边框、可调整大小，打开时到前方，之后遵守普通窗口层级；
- 设置面板：仍是详情窗口内部的渲染层面板，不额外创建第三个系统窗口。

## 2. 一个监控器如何服务两个窗口

文件监听器应该由主进程拥有，不能因为打开一个详情窗口就再创建一套监听器。否则会出现重复读取、重复事件、重复告警和难以清理的资源。

当前结构是：

```text
一个 CCMonitor
    -> 主进程广播目标
        -> 宠物窗口 webContents（告警气泡）
        -> 详情窗口 webContents（详情渲染）
```

广播前必须检查窗口和 `webContents` 是否已经销毁。详情窗口关闭后从广播目标中移除，但监控器继续为宠物窗口工作；只有宠物窗口关闭或应用退出时才停止监听。

## 3. 跨窗口操作为什么需要 IPC

渲染页面不能直接获得 `BrowserWindow`，否则会破坏 `contextIsolation` 和最小权限原则。宠物双击和详情关闭分别通过固定方法请求主进程：

```text
pet.js -> window.windowAPI.openDetail()
detail.js -> window.windowAPI.closeDetail()
```

`preload.js` 只暴露这两个明确动作，页面不能传入任意 IPC channel，也不能直接访问 Electron 或 Node.js。

可以用下面的规则判断应该使用事件总线还是 IPC：

| 通信范围 | 推荐方式 | 原因 |
|---|---|---|
| 同一个渲染页面内的模块 | `app.js` 事件总线 | 模块共享同一个 `window` 和 JavaScript 运行环境 |
| 两个 `BrowserWindow` 之间 | 主进程 + IPC | 每个窗口拥有独立的 DOM、`window` 和事件总线 |
| 渲染进程请求系统窗口能力 | preload 固定 API + IPC | `BrowserWindow` 只存在于主进程，页面不应直接获得 Electron 权限 |

不要为了方便向页面暴露 `send(channel, data)` 之类的通用接口。固定的 `openDetail()`、`closeDetail()` 更容易审计，也不会让页面自行构造任意 IPC channel。

## 4. 无边框窗口的拖动区域

去掉系统边框后，窗口不再自带标题栏，需要在 CSS 中声明拖动区域：

```css
.titlebar {
    app-region: drag;
    -webkit-app-region: drag;
    user-select: none;
}

.titlebar button {
    app-region: no-drag;
    -webkit-app-region: no-drag;
}
```

拖动区域会把鼠标操作交给系统窗口移动，因此按钮必须标记为 `no-drag`，否则关闭按钮会无法点击。当前同时保留标准写法和带前缀写法，以兼容项目使用的 Electron 版本。

显示器断开或排列改变时，主进程会检查详情窗口与所有显示器工作区的重叠面积。如果没有保留足够的可抓取区域，就把详情窗口恢复到主显示器中央。

## 5. 设置版本迁移与合并保存

设置结构从版本 1 升级为版本 2：

```json
{
  "version": 2,
  "projects": [],
  "petAlwaysOnTop": true
}
```

旧配置没有 `petAlwaysOnTop` 时自动使用 `true`，因此升级不会改变原有行为。写入项目列表时必须先读取现有设置再合并 `projects`，否则只保存项目字段会把置顶选项重置为默认值。

设置变化的数据流是：

```text
settings.js
    -> preload.js 固定 API
    -> ipcMain handler
    -> settings-store.js 原子保存
    -> petWindow.setAlwaysOnTop(...)
```

这样既保证修改立即生效，也保证应用重启后能够恢复。

### 为什么不能只保存正在修改的字段

当设置结构只有 `projects` 时，下面的代码看起来没有问题：

```js
saveSettings(settingsPath, { projects });
```

加入 `petAlwaysOnTop` 后，这样写会让保存对象缺少置顶字段。清洗函数只能把缺失字段恢复成默认值，于是用户每次增删项目都可能把已经关闭的置顶设置重新变成 `true`。

正确做法是“读取—合并—保存”：

```js
const currentSettings = loadSettings(settingsPath);
saveSettings(settingsPath, {
    ...currentSettings,
    projects,
});
```

以后增加移动速度、告警阈值或监控源选择时，也要检查每条写入路径是否保留了其他字段。

## 6. 生命周期检查清单

- 重复双击宠物只复用并唤回现有详情窗口；
- 关闭详情窗口不会停止监控或关闭宠物；
- 关闭宠物窗口时同时关闭详情，并只停止一次监控器；
- 新详情窗口加载后主动请求刷新，避免错过创建之前的数据；
- 所有 IPC 都保持固定方法与固定数据类型；
- 项目增删不会覆盖其他设置字段。

### `close` 和 `closed` 的区别

- `close`：窗口正在请求关闭，该过程可能被事件处理器阻止；
- `closed`：原生窗口已经真正销毁，旧引用不能再继续使用。

因此清空 `detailWindow` 引用应放在 `closed` 事件中。再次打开详情时还要同时检查引用是否存在、窗口是否已经 `isDestroyed()`，不能只检查变量是不是 `null`。

### 谁拥有监控器

当前宠物窗口是应用的主要生命周期所有者：关闭详情不停止监控，关闭宠物才停止监控并关闭详情。未来加入系统托盘后，应用即使隐藏宠物也可能继续运行，届时监控器的所有权应进一步上移到 `app` 或托盘生命周期，不能继续简单等同于宠物窗口生命周期。

## 7. 生命周期测试应该验证什么

Electron 测试不只是验证某个函数是否被调用，更重要的是验证对象之间的数量和所有权：

- 启动时只有一个宠物窗口和一个监控器；
- 重复打开详情不会产生多个详情窗口；
- 同一份监控数据能发送到所有存活窗口；
- 关闭详情不会调用 `stopWatching()`；
- 关闭宠物时监控器只停止一次；
- 修改项目列表不会覆盖置顶设置；
- 窗口完全离开显示器后能回到可见区域。

这类测试可以提前发现重复监听、资源泄漏、陈旧窗口引用和设置字段互相覆盖等问题。真实的窗口拖动、焦点层级和多显示器行为仍需要 Electron 手动验收，因为普通 Node 测试没有真正的操作系统窗口。

# 分层测试与 GitHub 验收

## 一、为什么一个功能需要多层测试

上下文告警的完整数据链路很长：

```text
Claude Code 对话
    -> statusLine
    -> claude-statusline-bridge.js
    -> data/cc-sessions/<session-id>.json
    -> cc-monitor.js
    -> preload.js
    -> app.js
    -> bubble.js
```

如果只看最后有没有气泡，失败时很难判断问题发生在哪一层。更合适的做法是把长链路拆成不同层次，分别验证。

## 二、第一层：单元测试验证状态转换

`tests/app-cc-flow.test.js` 不启动 Electron 窗口，而是把测试数据直接交给 `app.js`。

它适合快速验证：

- 首次观察会话时只建立基线；
- 79% 到 80% 会触发警告；
- 80% 到 81% 不会重复警告；
- 81% 到 95% 会触发危险提醒；
- 95% 到 96% 不会重复提醒；
- 降回 79% 后，再到 80% 可以重新提醒；
- 多个会话的状态不会互相覆盖；
- `null`、字符串和越界百分比不会产生错误告警。

这种测试速度快、结果稳定，适合覆盖大量边界条件。但它没有验证文件监听、Electron IPC 和真实 DOM。

## 三、第二层：临时快照验证 UI 数据链路

临时快照测试使用与 `claude-statusline-bridge.js` 相同的数据格式，在 `data/cc-sessions/` 中创建一个专用会话文件。

测试数据仍然经过正式链路：

```text
临时快照
    -> cc-monitor.js
    -> preload.js
    -> app.js
    -> bubble.js
```

因此它可以验证：

- `fs.watch()` 能否接收文件变化；
- `cc-monitor.js` 能否校验、合并和发布会话数据；
- IPC 是否把数据送到渲染进程；
- `app.js` 是否正确比较新旧状态；
- `bubble.js` 是否显示、定位并按时隐藏气泡。

它没有验证的是最前面的入口：

```text
Claude Code 对话 -> statusLine -> claude-statusline-bridge.js
```

这部分已经有自动化桥接测试，但真实 Claude Code 版本、配置和运行环境仍需要最终观察。

## 四、为什么临时快照不能覆盖真实会话

测试文件必须使用独立 UUID，例如：

```text
aaaaaaaa-aaaa-4aaa-8aaa-aaaaaaaaaaaa.json
```

文件名必须与 JSON 内部的 `sessionId` 一致，否则监控器会拒绝快照。独立 UUID 可以避免覆盖已有会话文件。

测试快照只应包含界面需要的白名单字段，不应写入：

- 提示词；
- 回复正文；
- transcript 内容或路径；
- API Key、Token 或其他凭据。

测试结束后需要：

1. 删除临时快照；
2. 确认测试 Electron 进程已关闭；
3. 确认真实快照仍然存在；
4. 运行 `git status`，确认测试数据没有进入版本控制。

`data/` 虽然已被 Git 忽略，但仍应主动清理测试文件，避免下次启动时把它当成历史会话。

## 五、第三层：真实对话测试验证入口

真实对话测试覆盖完整链路：

```text
Claude Code 对话
    -> Claude Code 调用 statusLine
    -> 桥接脚本写快照
    -> 桌宠显示告警
```

它能发现临时快照测试无法发现的问题，例如：

- 本机 `settings.json` 中的 statusLine 命令路径失效；
- Claude Code 更新后输入字段发生变化；
- 实际模型没有提供上下文窗口；
- 桥接脚本没有写入新快照；
- 仓库移动后外部配置仍指向旧路径。

但不应为了测试而在重要会话中故意消耗大量上下文。更合理的做法是：

- 用单元测试覆盖边界条件；
- 用临时快照完成可控 UI 验收；
- 在真实会话自然接近阈值时做最终观察。

真实对话观察可以作为上线后的确认项，不必阻塞已经被其他两层充分验证的状态机逻辑。

## 六、透明置顶窗口为什么可能截图漏判

桌宠窗口具有透明、无边框和置顶特性，截图中还会包含它后面的应用内容。气泡又是短暂元素，可能出现以下情况：

- 截图发生在文件监听处理之前；
- 气泡已经开始显示，但截图工具没有正确合成透明层；
- 宠物移动后气泡位置变化，截图观察区域不明显；
- 截图时气泡刚好结束隐藏。

因此 UI 验收不能只依赖一张截图。更可靠的证据组合是：

```text
自动化事件断言
    + 临时快照完整序列
    + 用户实时观察
    + 测试后的状态清理
```

如果截图与实时观察不一致，应先判断截图工具的限制，不应立即把“没截到”当成“没有发生”。

## 七、Issue 为什么不应在开发分支完成时立即关闭

开发分支上的代码还没有进入默认分支。此时即使测试通过，其他用户从 `main` 获取代码时仍然没有这个功能。

推荐节奏是：

```text
Issue 定义问题与验收标准
    -> 开发分支实现
    -> 自动化测试与手动验收
    -> 在 Issue 留进展评论
    -> 创建 Draft PR
    -> 整体审阅
    -> 合并到 main
    -> 关闭 Issue
```

Issue 评论应记录：

- 实现位于哪个分支和提交；
- 完成了哪些验收标准；
- 运行了哪些测试；
- 是否还有人工观察项；
- 为什么暂时保持 Open。

这样即使开发对话结束，后续查看 Issue 的人也能知道真实进度。

## 八、Draft PR 和 `Closes #7`

Draft PR 表示代码已经推送并可以查看，但还没有声明“可以立即合并”。它适合以下情况：

- 分支包含多个开发阶段，需要整体审阅；
- 功能已完成，但还要检查文档、提交范围或兼容性；
- 希望先建立 Issue 与代码之间的可追踪关系。

在 PR 描述中写：

```text
Closes #7
```

不会在创建 PR 时立刻关闭 Issue。只有 PR 合并进仓库默认分支后，GitHub 才会自动关闭对应 Issue。

如果 PR 被关闭但没有合并，Issue 仍然保持 Open。

## 九、连接器失败后的回退原则

GitHub 连接器可能具有读取权限，但没有评论或创建 PR 的写权限。典型表现是 GitHub 返回：

```text
403 Resource not accessible by integration
```

遇到这种情况应先确认操作没有部分成功，再使用已经认证的 `gh` CLI 完成同一个明确目标。

回退时需要保持：

- 仓库、Issue 或 PR 目标不变；
- 评论和 PR 内容不变；
- 不因工具变化扩大操作范围；
- 在执行 `push`、创建 PR 或其他远程写操作前仍然取得用户确认。

## 十、本轮学到的通用知识

```text
现象：单元测试通过后，仍然需要确认桌宠是否真的显示气泡。
原因：单元测试只验证局部逻辑，长数据链路中的文件监听、IPC 和 DOM 仍可能失败。
知识点：使用分层测试缩小问题范围，并让每一层提供不同类型的证据。
```

```text
现象：开发分支已经完成，但 Issue 仍保持 Open。
原因：默认分支还没有获得该功能。
知识点：Issue 表示交付状态，PR 表示待合并变更；合并是两者之间的完成边界。
```

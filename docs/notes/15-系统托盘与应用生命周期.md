# 系统托盘与应用生命周期

## 为什么加入托盘后要重新划分所有权

没有系统托盘时，宠物窗口是用户看到的唯一入口。原来的实现因此让宠物窗口同时拥有 `CCMonitor`：创建宠物时启动监控，宠物关闭时停止监控。

加入托盘后，应用即使没有可见窗口也应该继续运行。此时更合理的所有权是：

```text
Electron app
├── Tray
├── CCMonitor
├── petWindow
└── detailWindow
```

`app` 是生命周期最长的所有者。托盘和监控器都跟随应用启动与退出；宠物窗口和详情窗口只是可以显示、隐藏或销毁的界面。

这样做的直接效果是：通过托盘隐藏宠物不会停止 Claude Code 监控，关闭详情也不会结束应用。用户从托盘选择“退出应用”，或直接关闭宠物窗口时，监听器才会释放。

## `hide`、`close`、`destroy` 和 `quit`

这些操作看起来都能让窗口消失，但含义不同：

- `win.hide()`：隐藏原生窗口，`BrowserWindow` 和渲染进程仍然存在，可以再次 `show()`；
- `win.close()`：发起一次正常关闭请求，会触发 `close`，监听器可以调用 `event.preventDefault()` 阻止关闭；
- `win.destroy()`：直接销毁原生窗口，不应当用来表达普通隐藏；
- `app.quit()`：请求整个 Electron 应用退出，会依次关闭窗口并进入应用退出事件。

“隐藏”和“关闭”采用不同入口表达不同意图：

- 托盘菜单的“隐藏桌宠”只调用 `hide()`，应用和监控器继续运行；
- 用户对宠物窗口按 `Alt+F4` 或发出正常关闭请求，窗口真正关闭，并由 `closed` 事件继续调用 `app.quit()`；
- `before-quit` 把 `isQuitting` 设为 `true`，用于防止退出过程再次触发重复的 `app.quit()`。

详情窗口仍采用真正关闭。因为它可以由双击宠物或托盘菜单重新创建，而且关闭详情不影响应用级监控器。

## `window-all-closed` 为什么不再退出应用

普通 Electron 示例通常在 Windows 和 Linux 上监听 `window-all-closed` 并调用 `app.quit()`。这适合“窗口就是应用”的程序，但不适合需要托盘常驻的后台工具。

本项目现在保留 `window-all-closed` 监听器，但不把“所有辅助窗口都已关闭”统一解释成退出。托盘仍然提供显示桌宠、打开详情、打开设置和退出应用的入口；宠物窗口关闭则由自身的 `closed` 事件明确请求退出。

## 宠物窗口为什么不显示在任务栏

宠物本体是后台工具的角色界面，不应长期占据普通应用的任务栏按钮。创建宠物 `BrowserWindow` 时使用：

```js
skipTaskbar: true
```

Electron 在 Windows 和 macOS 上用该选项控制窗口是否出现在任务栏。它只隐藏任务栏入口，不会结束 Electron 进程，也不会让应用从任务管理器中消失。

详情和设置属于用户主动打开的工作界面，因此详情窗口明确使用 `skipTaskbar: false`，保持可切换、可最小化和可找回。不能把宠物窗口的任务栏策略机械复制给所有窗口。

真正的清理发生在应用退出阶段：

```text
托盘“退出应用”
    -> app.quit()
    -> before-quit
        -> isQuitting = true
        -> CCMonitor.stopWatching()
    -> 关闭窗口
    -> will-quit
        -> Tray.destroy()
```

`stopMonitor()` 本身是幂等的：监控器已经为空时直接返回。这样可以避免多个退出事件导致监听器重复释放。

## 托盘菜单为什么根据状态重新构建

菜单里的“显示桌宠 / 隐藏桌宠”和“桌宠始终置顶”不是固定文字：

- 宠物可见时显示“隐藏桌宠”；
- 宠物隐藏时显示“显示桌宠”；
- 置顶项目的勾选状态来自持久化设置。

每次相关状态变化后重新调用 `Menu.buildFromTemplate()`，再用 `tray.setContextMenu()` 设置新菜单。菜单命令只调用主进程中的窗口和设置函数，不复制第二套业务逻辑。

当前菜单顺序保持稳定：

1. 显示或隐藏桌宠；
2. 打开详情；
3. 打开设置；
4. 桌宠始终置顶；
5. 退出应用。

“暂停监控”和“关于”尚未确定产品行为，因此没有为了填满菜单而提前加入。

## 从托盘打开设置的固定 IPC 链路

设置面板目前位于详情窗口内部，不是独立 `BrowserWindow`。托盘菜单在主进程运行，不能直接操作详情页面的 DOM。因此使用一条固定通知：

```text
Tray 菜单
    -> showDetailWindow({ openSettings: true })
    -> detailWindow.webContents.send("settings:open")
    -> preload.js 的 onOpenSettings(callback)
    -> settings.js 的 showSettings()
```

如果详情窗口已经存在，就复用、显示并聚焦原实例；如果尚未创建，则等待 `ready-to-show` 后再发送通知。

`preload.js` 只暴露 `onOpenSettings()`，没有暴露任意 channel 的通用 `on()` 或 `send()`。这继续遵守最小权限原则：渲染页面只能订阅经过审核的固定能力。

## 图标路径与安全降级

托盘从以下稳定路径读取草稿图标：

```js
path.join(__dirname, "assets", "tray", "tray-icon-32.png")
```

如果文件缺失，主进程会记录错误并使用 `nativeImage.createEmpty()` 创建安全回退，避免图标资源错误阻止窗口和监控器启动。空图像不代表理想体验，但它把“资源显示失败”限制在托盘层，不扩散成整个应用崩溃。

后续打包工作仍要确认 `assets/tray/` 被包含在发行文件中。Windows 官方建议使用 `.ico` 获得最佳托盘效果；当前 PNG 是 MVP 草稿，真实主题、缩放和打包效果需要继续手动验收。

## 托盘单击为什么只负责唤起桌宠

Windows 通知区域交互指南建议左键单击提供可见反馈，例如打开最常用的窗口、状态浮层或菜单；左键双击通常执行默认命令。Electron 的 `Tray` 同时提供 `click` 和 Windows 可用的 `double-click` 事件，但没有提供 Windows 用户配置的系统双击时间。

如果同时给单击和双击绑定不同命令，常见做法是延迟单击，等待一段固定时间确认它不是双击。这会产生两个问题：

- 固定延迟未必与用户在 Windows 中设置的双击速度一致；
- 单击响应被人为变慢，快速操作还可能执行错误命令或造成窗口闪烁。

因此当前映射保持简单且符合主流预期：

- 左键单击：显示并唤起桌宠；
- 左键双击：暂不绑定；
- 右键：打开现有原生快捷菜单。

单击调用的是幂等的 `showPetWindow()`。无论连续点击多少次，它都复用同一个 `petWindow`，不会创建重复窗口或监控器。唤起顺序为：

```text
检查窗口是否仍可用
    -> 检查与现有显示器工作区是否有足够重叠
    -> 不可见时移动到鼠标所在显示器中央
    -> 最小化时 restore()
    -> show()
    -> focus()
    -> moveTop()
```

`show()` 和 `focus()` 让隐藏窗口重新可见并请求焦点；`moveTop()` 只执行一次 Z-order 前置。这里不能调用 `setAlwaysOnTop(true)`：永久置顶是用户偏好，托盘唤起只是一次操作。如果为了“找回”而打开永久置顶并保存设置，用户重启应用后仍会受到一次临时操作的影响。

窗口可见性不能只检查左上角坐标。实现会计算窗口与所有显示器工作区的重叠宽度和高度；至少保留一块可操作区域才认为还能找回。若显示器已断开或窗口完全落在工作区外，则根据 `screen.getCursorScreenPoint()` 选择用户当前所在显示器，并把桌宠移动到该工作区中央。

参考资料：

- [Microsoft：Notification Area 交互指南](https://learn.microsoft.com/en-us/windows/win32/uxguide/winenv-notification)
- [Electron：Tray API](https://www.electronjs.org/docs/latest/api/tray/)
- [Electron：BrowserWindow API](https://www.electronjs.org/docs/latest/api/browser-window)

## 为什么应用单例不能只靠 `createTray()`

`createTray()` 中的 `if (tray) return tray` 只能防止同一个 JavaScript 进程重复创建托盘。如果用户再次运行应用，Windows 会启动另一个 Electron 主进程；新进程有自己的内存，因此它看到的 `tray` 和 `ccMonitor` 仍然都是 `null`。

应用需要在创建窗口、托盘、IPC 和监控器之前申请进程级锁：

```js
if (!app.requestSingleInstanceLock()) {
    app.quit();
    return;
}
```

拿不到锁说明已有主实例。第二实例立即退出模块初始化，不加载监控器，也不注册后续生命周期。Electron 同时会在主实例中触发 `second-instance`；本项目继续调用幂等的 `showPetWindow()`，复用桌宠并将其恢复到前方。

```text
第二次启动
    -> requestSingleInstanceLock() 返回 false
    -> 通知主实例 second-instance
        -> showPetWindow()
    -> 第二实例 app.quit()
    -> 不创建窗口、Tray 或 CCMonitor
```

单实例锁解决的是“跨进程只有一个资源所有者”；`createTray()`、`createPetWindow()` 和 `startMonitor()` 内部的幂等判断解决的是“同一进程内避免重复创建”。两层保护不能互相替代。

参考资料：[Electron：app 单实例锁与 `second-instance`](https://www.electronjs.org/docs/latest/api/app#apprequestsingleinstancelockadditionaldata)。

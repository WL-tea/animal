# 上下文阈值与状态跨越

## 一、看到的现象

Claude Code 的 `statusLine` 快照会随着会话继续输入而反复更新。如果每次收到数据时都判断“是否大于 80%”，那么一个保持在 81% 的会话会连续弹出相同气泡。

真正需要提醒的不是“当前仍然高于阈值”，而是“刚刚进入了更高的告警区间”。

## 二、把百分比分成状态

当前使用三个区间：

```text
normal:  0% <= 使用率 < 80%
warning: 80% <= 使用率 < 95%
danger:  95% <= 使用率 <= 100%
```

每次收到有效百分比时，先把数值转换成状态，再比较新旧状态：

```text
normal  -> warning：提醒接近上限
warning -> danger：提醒危险
warning -> warning：不重复提醒
danger  -> danger：不重复提醒
danger  -> normal：只更新状态，不提醒
normal  -> warning：下降后再次越界，可以重新提醒
```

这类只在状态发生特定变化时执行动作的方式，可以称为“边沿触发”。与它相对的是“电平触发”：只要条件持续成立，每次检查都会执行动作。

## 三、为什么要按会话保存状态

同一个项目可能同时存在多个 Claude Code 会话。一个会话是 96%，另一个会话是 20% 时，只看项目最新会话会漏掉危险状态。

因此告警状态使用下面的组合键：

```text
项目路径 + 会话 ID
```

这样每个会话独立经历 `normal`、`warning` 和 `danger`，不会互相覆盖。

## 四、首次数据为什么只建立基线

应用启动后会读取磁盘上已有的会话快照。这些快照可能早已处于 80% 或 95%，并不代表刚刚发生了阈值跨越。

第一次观察某个会话时只记录其当前状态，不显示气泡。后续收到该会话的新数据，且状态进入更高区间时，才触发提醒。这样可以避免每次启动桌宠都重放历史告警。

## 五、无效数据为什么不能当成 0%

文件读取失败、快照缺少 `context` 或百分比格式错误时，真实使用率是“未知”，不是 0%。

如果把未知值当成 0%，状态会错误地从 `danger` 降到 `normal`；下一次恢复到 95% 时又会重复告警。因此无效数据应直接忽略，并保留上一次有效状态。

## 六、本轮数据流与职责

```text
cc-monitor
    -> preload
    -> app.js 收到数据并发出 cc:update
    -> app.js 比较每个会话的新旧告警状态
    -> 必要时发出 bubble:say
    -> bubble.js 负责气泡 DOM、定位和隐藏计时
```

`app.js` 负责页面级状态和事件协调，但不操作气泡 DOM；`bubble.js` 只负责显示，不理解上下文阈值。判断逻辑和展示逻辑分开后，测试可以直接检查事件，不需要启动完整 Electron 窗口。

# Windows 系统托盘图标与资源路径

## 为什么托盘图标需要专用资源

系统托盘留给每个应用的显示面积很小。普通窗口图标或大尺寸插画即使在原图中很精细，缩小到 16～24 像素后也可能糊成色块。托盘图标更需要清楚的外轮廓、较大的关键特征和足够的透明留白。

Windows 还会根据显示缩放比例选择或缩放图像。100%、125%、150%、175% 和 200% 缩放环境并不总是使用同一个物理像素尺寸，因此只准备一张 16px 图片会导致放大模糊，只准备一张大图又可能在缩小时丢失细节。

托盘图标与以下资源的职责不同：

- 托盘图标：强调小尺寸辨识度和浅色、深色任务栏适配；
- 窗口图标：用于普通窗口、Alt+Tab 等较大的系统界面；
- 安装包和可执行文件图标：通常需要 `.ico` 内包含多种尺寸；
- 宠物皮肤：属于可替换的角色内容，不应直接决定应用的固定系统入口。

九张派生图片都来自同一张母图。这样可以保证颜色、构图和透明边缘一致，也能避免不同尺寸被分别修改后逐渐变成九套不同图标。

母图是视觉来源，派生文件是运行时可以直接选择的交付物。后续替换草稿时，不能只修改 `tray-icon-source.png` 而保留旧的派生文件，必须重新导出全部尺寸并重新验证。

## 透明通道与小尺寸可读性

PNG 的 alpha 通道记录每个像素的不透明程度：

- alpha 为 0：完全透明；
- alpha 为 255：完全不透明；
- 中间值：用于抗锯齿边缘，让曲线不会出现明显台阶。

本次图标四角完全透明，主体周围保留安全边距。圆脸、短耳朵和双眼使用较大的几何形状，避免依赖缩小后会消失的细线。

自动检查确认每张派生图：

- 宽高与文件名中的尺寸一致；
- 图像模式为 RGBA；
- 四角透明；
- 可见像素覆盖率处于合理范围。

自动检查不能替代肉眼检查。16px 图标即使通过尺寸和透明度校验，也可能看不清眼睛或耳朵。因此还需要把各尺寸分别放到浅色和深色背景中预览，检查轮廓、关键特征和色边。

## 开发路径与打包路径

本次 Issue #28 只建立资源，没有修改 Electron 主进程。后续 #29 创建 `Tray` 时，应该在主进程从稳定路径读取资源，例如基于 `__dirname` 组合仓库内路径：

```js
path.join(__dirname, "assets", "tray", "tray-icon-32.png")
```

`__dirname` 表示当前模块所在目录；它不会随着用户从哪个终端目录启动应用而变化。相反，直接依赖 `process.cwd()` 容易在开发启动、快捷方式启动和打包运行之间得到不同结果。

打包工具还需要明确把 `assets/tray/` 放入最终应用。开发环境能读取文件，不代表打包后一定存在。#29 接入时应同时验证开发版和打包版路径，并为资源缺失提供安全降级，不能让一个图标文件错误阻止整个应用启动。

生成过程中的 `C:/Users/lenovo/.codex/generated_images/` 和 `tmp/imagegen/` 只是本机临时位置，运行时代码不得引用它们。

## 如何替换当前草稿

当前 A 版图标只是可用草稿。未来替换时应遵循以下顺序：

1. 保持正方形构图、透明背景和足够安全边距；
2. 替换 `tray-icon-source.png`；
3. 从新母图重新导出 16、20、24、32、40、48、64、128、256px；
4. 验证 RGBA、透明四角、像素尺寸和主体覆盖率；
5. 在浅色、深色背景中人工检查 16px 起的所有常用尺寸；
6. 在真实 Windows 托盘中重新截图验收。

不要分别手工修补九张派生图片，也不要把用户选择的宠物皮肤直接复制为托盘图标。应用系统入口需要稳定、可识别；皮肤资源则可能具有任意比例、透明边界和视觉复杂度。

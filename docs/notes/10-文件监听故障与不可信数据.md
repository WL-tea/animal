# 文件监听故障与不可信数据

## 一、看到的现象

statusLine 快照监听已经处理了 `fs.watch()` 的 `error` 事件，但旧 backup 监听没有相同保护。

同时，backup JSON 中的费用、Token 和时长字段被直接发送到渲染层。如果 `lastCost` 变成字符串，详情面板调用 `toFixed()` 时可能抛出异常。

这两个问题看起来分别属于“文件监听”和“页面渲染”，根本原因却相同：

```text
程序信任了来自自身进程之外的数据和状态。
```

## 二、fs.watch 为什么需要 error 监听

`fs.watch()` 返回的是 EventEmitter。目录权限变化、磁盘故障、网络盘断开或操作系统监听资源异常时，watcher 可能发出 `error`。

如果 EventEmitter 的 `error` 没有监听器，Node.js 会把它当成未处理异常。这不仅会让某次更新失败，还可能影响整个 Electron 主进程。

因此 watcher 建立后需要立即注册：

```js
watcher.on("error", (err) => {
    // 记录、关闭、清空引用并降级
});
```

创建 watcher 本身也可能同步抛错，所以 `fs.watch()` 还需要放在 `try/catch` 中。

## 三、为什么出错后要设为 null

watcher 出错后已经不再可信。只调用 `close()` 但保留对象引用，会让后续代码误以为监听仍然存在：

```js
if (this.backupWatcher) {
    // 错误地认为不需要恢复
}
```

正确状态转换是：

```text
有效 watcher
    -> error
    -> close()
    -> backupWatcher = null
```

后续主动刷新看到 `null` 后，才能重新调用 `startBackupWatching()` 安装替代 watcher。

## 四、为什么错误处理要确认 watcher 身份

错误监听器会闭包保存它所属的 watcher：

```js
const watcher = fs.watch(...);
```

处理错误前还要判断：

```js
if (this.backupWatcher !== watcher) return;
```

如果旧 watcher 在替代 watcher 已经安装后才迟到地发出错误，它不应关闭新的 watcher。这个身份判断可以防止旧异步事件破坏新状态。

## 五、解析失败和监听失效采用不同降级

单个 backup 文件解析失败时，可能只是写入尚未完成或最新文件暂时损坏。此时保留上一次已经验证的数据，比立即清空界面更稳定：

```text
单个文件解析失败 -> 记录错误 -> 保留上一次有效数据
```

但目录无法枚举或 watcher 整体失效时，程序已经不能确认累计数据是否还会更新。继续展示旧数据容易让用户误以为它仍是实时值：

```text
目录或监听整体失效 -> 清空 backup 累计数据 -> 保留其他独立数据源
```

这不是唯一可用策略，但必须明确区分“最后一次已知值”和“仍然实时的值”。当前界面没有专门的 stale 标识，因此选择清空不再可信的累计数据。

## 六、数据源之间要独立降级

当前项目有两类 Claude Code 数据：

```text
backup：累计费用、累计 Token、模型统计
statusLine 快照：当前会话、上下文窗口、实时百分比
```

backup watcher 故障时，不应清空 `sessionSnapshots` 或停止 statusLine watcher。否则一个兼容数据源失败，会把更可靠的实时上下文一起拖垮。

目标状态是：

```text
backup 故障
    -> 累计费用数据降级
    -> statusLine 会话继续发布
    -> 详情上下文和告警继续工作
```

这种设计可以称为故障隔离：每个数据源独立启动、失败、恢复和停止。

## 七、本机文件为什么仍是不可信输入

“文件在自己电脑上”不等于“结构永远正确”。本机 JSON 仍可能因为以下原因异常：

- 写入只完成一半；
- 软件版本升级后字段类型变化；
- 用户手动编辑；
- 旧文件使用不同结构；
- 磁盘或同步软件造成内容损坏；
- 测试数据没有及时清理。

因此数据进入系统边界时就应规范化，而不是等页面调用 `toFixed()` 时才发现类型不对。

## 八、为什么不自动转换字符串数字

下面的值看起来都能转换：

```js
Number("2.5")
Number("")
Number(null)
```

但结果不一定符合业务含义。空字符串和 `null` 都可能变成 0，从而掩盖上游格式错误。

当前规范是只接受：

```text
typeof value === "number"
Number.isFinite(value)
value >= 0
```

不满足条件就降级为 0。这样数据契约更明确：字符串即使长得像数字，也不是合法数值字段。

## 九、对象字段也需要验证

JavaScript 中数组和 `null` 的 `typeof` 都可能让简单判断出错：

```js
typeof null === "object"
typeof [] === "object"
```

因此普通对象判断需要同时排除它们：

```js
value !== null
    && typeof value === "object"
    && !Array.isArray(value)
```

`lastModelUsage` 先确认是普通对象，再逐个规范化模型的输入 Token、输出 Token 和费用。异常模型项被跳过，不让它破坏整个项目统计。

## 十、入口规范化与稳定数据契约

推荐的数据流是：

```text
原始文件
    -> 解析
    -> 类型和范围校验
    -> 生成稳定内部结构
    -> 发送给渲染层
    -> 页面只负责格式化和显示
```

这样渲染层可以依赖明确的数据契约，不需要在每个 DOM 模板里重复判断字符串、负数、数组和 `null`。

它不能取代渲染层的 HTML 转义；数值结构安全与字符串注入防护是两个不同问题，应分别处理和测试。

## 十一、本轮学到的通用知识

```text
现象：一个文件监听器失败可能影响整个 Electron 主进程。
原因：EventEmitter 的 error 事件没有监听器，并且失效 watcher 仍被当成有效对象。
知识点：长期资源必须拥有完整的创建、错误、清理和恢复生命周期。
```

```text
现象：本机 JSON 中的字符串费用让详情渲染可能抛错。
原因：原始外部数据直接穿过主进程进入 UI。
知识点：在数据入口建立稳定契约；文件位置可信不代表文件内容可信。
```
